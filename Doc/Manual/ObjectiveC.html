This chapter describes SWIG's Objective-C module. The module generates Objective-C proxy classes and Objective-C++ wrapper code for calling existing C and C++ libraries from Objective-C applications.

Objective-C is currently an experimental target language. The module supports the core SWIG wrapping model, examples and test-suite coverage described here, but it should continue to mature before being treated as a fully supported target with long-term backwards compatibility guarantees.

The generated code is intended for manual reference-counted Objective-C code. It can be used from Automatic Reference Counting projects by compiling the generated proxy implementation files with ARC disabled.

41.1 Overview

Objective-C support is selected with the -objc option. The default mode generates both the C/C++ wrapper layer and an Objective-C proxy layer. The proxy layer exposes C functions, C variables, constants, enums, structures, C++ classes, inheritance and selected standard library types as Objective-C interfaces.

The Objective-C module supports directors for cross-language polymorphism. A C++ virtual method can call back into an Objective-C subclass when the class is wrapped with director support enabled.

41.2 Examples

Examples are provided in the SWIG source tree under Examples/objc. The simple example demonstrates C function and variable wrapping. The class example demonstrates C++ class wrapping. Additional examples cover callbacks, constants, enums, pointers, references, templates and global variables.

41.3 Running SWIG with Objective-C

For a C interface file: 

$ swig -objc example.i

For a C++ interface file: 

$ swig -c++ -objc example.i

41.3.1 Objective-C command line options

The Objective-C-specific options are:

Objective-C specific options
-noproxy Do not generate Objective-C proxy files. Only the lower-level wrapper files are generated.

These options can also be shown by running: 

$ swig -objc -help

41.3.2 Generated files

For a module named example, SWIG normally generates these files:

File Description
example_wrap.h C declarations for the lower-level wrapper interface.
example_wrap.mm Objective-C++ implementation of the lower-level wrapper interface.
example_proxy.h Objective-C declarations for the generated proxy API.
example_proxy.mm Objective-C++ implementation of the generated proxy API.

When -noproxy is used, only the lower-level wrapper files are generated.

41.3.3 Compiling a module

On macOS, the generated wrapper and proxy implementation files can be compiled with an Objective-C compiler and linked with Foundation: 

$ swig -c++ -objc example.i
$ clang++ -c example_wrap.mm example_proxy.mm -I.
$ clang++ -dynamiclib example_wrap.o example_proxy.o -framework Foundation -o libexample.dylib
$ clang -x objective-c runme.m -I. -L. -lexample -framework Foundation -o runme

When the wrapped library is written in C rather than C++, the generated wrapper file still uses the .mm extension and can be compiled as Objective-C. This lets the proxy implementation and the wrapper implementation share the same Objective-C runtime support.

41.4 A tour of basic C/C++ wrapping

41.4.1 Functions

C global functions are emitted as Objective-C functions in the proxy layer. To avoid collisions with ordinary C function names, non-accessor proxy functions are prefixed with Objc.

41.4.2 Global variables

Global variables are exposed through getter and setter proxy functions. For a variable named value, the proxy layer emits getValue() and setValue(...) where the input variable is mutable. Variables marked with %immutable only receive a getter.

41.4.3 Constants and enums

Simple numeric constants and enums are emitted into the proxy API. Constants which cannot be represented as Objective-C compile-time initializers are exposed through generated accessor functions.

41.4.4 Structures and C++ classes

Structures and C++ classes are represented by Objective-C proxy classes. Each proxy object stores the underlying C/C++ pointer and an ownership flag. Public constructors are mapped to Objective-C initializers and public methods are mapped to instance or class methods.

41.4.5 Memory management

Proxy objects use manual reference counting. A proxy owns the C/C++ object when it is created by a generated constructor or when a typemap explicitly transfers ownership. The generated dealloc method deletes the underlying C/C++ object only when the proxy owns it.

The generated proxy classes also provide swigRelease methods which detach ownership from the Objective-C proxy and return the raw C/C++ pointer.

41.4.6 Inheritance

The Objective-C module mirrors C++ single inheritance using Objective-C inheritance where possible. For multiple C++ bases, the generated proxy class stores additional base-class pointers and emits conversion accessors, because Objective-C classes can have only one superclass.

41.4.7 Templates

Template instantiations are wrapped using the standard SWIG template declarations. The generated Objective-C proxy class names are derived from the SWIG symbol names, so %rename can be used to choose Objective-C API names for template specializations.

41.4.8 STL and smart pointers

The Objective-C library includes support files for common C++ standard library types including std::string, std::wstring, std::vector, std::deque, std::map and std::pair. The library also includes support for std::auto_ptr and boost::shared_ptr.

41.4.9 Directors

Directors are enabled in the usual SWIG way with the -directors command-line option and %feature("director") on the classes which need Objective-C overrides. 

%module(directors="1") example
%feature("director") Base;

When a director-enabled C++ virtual method is called, the director checks whether the Objective-C object overrides the corresponding selector. If it does, the call is dispatched to the Objective-C method; otherwise the C++ base implementation is used.

41.5 Objective-C typemaps

The Objective-C module uses a family of typemaps to translate between C/C++ types, wrapper function types and Objective-C proxy types.

Typemap Purpose
imtype The lower-level wrapper function type.
objctype The public Objective-C proxy type.
in Convert wrapper arguments to C/C++ arguments.
out Convert C/C++ return values to wrapper return values.
objcin Convert Objective-C proxy values to wrapper argument values.
objcout Convert wrapper return values to Objective-C proxy values.
objcdirectorin Convert C++ director callback arguments to Objective-C values.
objcdirectorout Convert Objective-C director callback return values to C++ values.

Additional class customization typemaps such as objcbase, objcprotocols, objcinterfacecode, objcimplementationcode and objcdestructor can be used to customize generated proxy classes.

41.6 Foundation headers and naming

Objective-C user code commonly imports Foundation before importing generated proxy headers: 

#import <Foundation/Foundation.h>
#import "example_proxy.h"

The generated proxy headers avoid importing Foundation directly. This keeps common C and C++ identifiers from being accidentally hidden by Foundation or system C header macros. Instead, the proxy headers use lightweight forward declarations for Foundation classes such as NSString, NSArray, NSData and NSException.

Some Objective-C and Foundation names are reserved by the Objective-C backend. If user code wraps a C/C++ symbol with a reserved name, SWIG renames the generated Objective-C symbol and emits the standard keyword warning.
