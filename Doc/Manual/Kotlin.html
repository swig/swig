This chapter describes SWIG's support of Kotlin on the JVM. The Kotlin module generates Kotlin/JVM code that calls into the C/C++ wrapper code through the Java Native Interface (JNI), so it shares its overall architecture and most of its JNI infrastructure with the Java module.

The Kotlin module is an experimental target language, see Target languages.

29.1 Overview

The Kotlin module works in the same three layer way as the Java module:

  1. A C/C++ wrapper file (example_wrap.cxx) containing the JNI functions.
  2. An intermediary Kotlin object (exampleJNI.kt) containing external fun declarations for each JNI function.
  3. Kotlin proxy classes, one per C++ class, plus a module object for global functions, variables and constants.

Unlike the Java module no Java source files are generated at all and only the Kotlin compiler is needed to compile the generated code. The intermediary object compiles to a JVM class of the same name, so the JNI functions in the C/C++ wrapper file resolve against it directly.

A simple example. Given the interface file: 

/* File: example.i */
%module example

%inline %{
int gcd(int x, int y);
extern double Foo;
%}

SWIG is invoked with the -kotlin option: 

$ swig -kotlin -c++ example.i

and the generated code can be used from Kotlin as follows: 

fun main() {
    System.loadLibrary("example")
    println(example.gcd(42, 105))
    example.Foo = 3.1415926      // global variables become properties
    println(example.Foo)
}

29.2 Differences to the Java module

The generated Kotlin code aims to be idiomatic Kotlin rather than a literal transliteration of the generated Java code:

  • Global functions, variables and constants are placed in a module level object instead of a class with static methods. There is no separate constants interface - constants are val declarations in the module object.
  • C++ member variables are wrapped as Kotlin properties with custom get() and set(value) accessors instead of getter and setter methods. Read only variables become val properties. Static member variables become properties in the proxy class companion object.
  • Static member functions and constants are placed in a single companion object within the proxy class, together with the getCPtr and swigRelease helpers.
  • Kotlin classes are final by default, so proxy classes are emitted as open class and the proxy methods for C++ virtual functions are emitted with the open modifier (or override when they override a method from a base proxy class).
  • Pointer types are mapped to nullable Kotlin types (for example Foo?), with a C++ NULL pointer corresponding to Kotlin null. References and pass by value types use the corresponding non-nullable types.
  • No finalizers are generated as Kotlin has no finalizer syntax and JVM finalization is deprecated. Memory is reclaimed by calling delete() explicitly or by registering proxy instances with java.lang.ref.Cleaner.
  • Kotlin has no checked exceptions, so no throws clauses are generated.

The directives of the Java module are available under Kotlin specific names:

Kotlin moduleJava module equivalent
%kotlinconst%javaconst
%kotlinconstvalue%javaconstvalue
%kotlinenum%javaenum
%kotlinmethodmodifiers%javamethodmodifiers
%kotlinexception%javaexception
%nokotlinexception%nojavaexception
%clearkotlinexception%clearjavaexception

Likewise, the typemaps of the Java module are available under Kotlin specific names. The jni typemap and all of the C/C++ side typemaps (in, out, directorin, directorout, throws etc.) are identical to the Java module. The Java side typemaps have Kotlin equivalents: ktype (was jtype), kstype (was jstype), kin, kout, kbody, kcode, kimports, kbase, kinterfaces, kclassmodifiers, kconstruct, kdestruct, kdirectorin and kdirectorout. The additional kcompanion typemap contains the members of the proxy class companion object. The special variable $kotlinclassname replaces $javaclassname and $kotlininput and $kotlincall replace $javainput and $javacall.

29.3 Wrapping C++ classes

C++ classes are wrapped by Kotlin proxy classes. Given the C++ code: 

class Shape {
public:
  double x;
  static int nshapes;
  virtual ~Shape();
  virtual double area();
  void move(double dx);
  static int count();
};

class Circle : public Shape {
public:
  Circle(double r);
  virtual double area();
};

the wrapped classes can be used naturally from Kotlin: 

val c = Circle(2.0)
c.x = 1.5                       // member variable as property
val s: Shape = c                // upcast to base proxy class
println(s.area())               // virtual dispatch through C++
Shape.nshapes                   // static member via companion object
c.delete()                      // explicit destruction of the C++ object

29.4 Enums

The four enum wrapping approaches of the Java module are supported and selected the same way by including one of enums.swg (proper Kotlin enum class), enumtypesafe.swg (the default, typesafe enum pattern), enumtypeunsafe.swg (constant integers in a class) or enumsimple.swg (plain constant integers). The proper enum approach generates an idiomatic Kotlin enum class where each enum constant carries its C++ value in the swigValue property: 

enum class Color {
  RED(exampleJNI.RED_get()),
  GREEN,
  BLUE(exampleJNI.BLUE_get());
  val swigValue: Int
  ...
}

29.5 Directors

Cross language polymorphism is supported via directors in the same way as the Java module. Director enabled proxy classes and their virtual methods are emitted with the open modifier so they can be overridden from Kotlin with the usual override keyword: 

class KotlinCallback : Callback() {
    override fun run(x: Int): Int = x * 3
}

C++ code invoking the virtual method through a base class pointer or reference then calls back into the Kotlin override.

29.6 STL support

The usual STL library files are provided (std_string.i, std_vector.i, std_map.i, std_list.i, std_set.i, std_pair.i, std_array.i, std_shared_ptr.i, std_unique_ptr.i and the unordered containers). The containers integrate with the Kotlin collection interfaces by extending the matching java.util abstract collection classes, so a wrapped std::vector is a MutableList, a wrapped std::map is a MutableMap and so on: 

val v = IntVector()
v.add(1)
v[0] = 2                        // MutableList operator access
for (n in v) println(n)         // Kotlin iteration
val m = StringIntMap()
m["key"] = 42
for ((k, n) in m) println("$k=$n")

29.7 Compiling and running

All the generated Kotlin files for a module should be compiled in a single kotlinc invocation. The compiled code is run on a JVM with the Kotlin runtime on the classpath, most easily via the kotlin launcher that ships with the Kotlin compiler: 

$ swig -kotlin -c++ example.i
$ g++ -c -fPIC example.cxx example_wrap.cxx -I"$JAVA_HOME/include" -I"$JAVA_HOME/include/linux"
$ g++ -shared example.o example_wrap.o -o libexample.so
$ kotlinc *.kt runme.kt -d .
$ kotlin -Djava.library.path=. -classpath . RunmeKt

As with Java, the native library is loaded with System.loadLibrary("example"), which can conveniently be placed in the init block of an object or at the start of main. See the Dynamic linking problems section in the Java chapter for help with library loading problems - the same advice applies to Kotlin.
