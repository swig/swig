40.1 Introduction

The Rust module is an experimental target language module for generating Rust bindings to C and C++ code. It is intentionally modelled on the structure of the C# module: SWIG emits a C or C++ wrapper source file, a raw Rust FFI layer, a public proxy layer, runtime support code, and optional director callback plumbing. The generated Rust API is expected to evolve while the module is reviewed and hardened, so no backwards compatibility guarantee is made while it has experimental status.

The generated Rust module directory is named after the SWIG module by default. It contains mod.rs, ffi.rs, runtime.rs, proxy.rs and module.rs. Wrapped C++ classes and top-level enums are emitted into their own sibling files such as class_Shape.rs and enum_Color.rs, then re-exported from mod.rs. When %nspace is enabled, namespace submodules are created below the generated module directory. This keeps the raw C ABI calls, shared runtime helpers, module-level functions and proxy types separate in the same broad style as the C# target's intermediary and proxy files.

The public Rust API wraps C++ classes as proxy structs, exposes constructors, destructors, member functions, static members, variables and constants, maps nullable pointer returns to Option, maps many proxy parameters to &T, &mut T, Option<&T> or Option<&mut T>, and supports Result-based C++ exception handling. C++ enums are generated as Rust #[repr(C)] enums with conversion helpers. Standard library interface files provide Rust-friendly helpers for strings, vectors, maps, sets, arrays, optional values and smart pointers.

The module also supports C++ directors. Directors let a C++ virtual method call dispatch back into Rust by implementing a generated Rust trait. The generated director helpers keep Rust callback state alive, provide explicit disconnect and disown operations, bridge Rust panics or structured callback errors back through the C++ director layer, and include convenience helpers for factory-style APIs which store a callback object and return another proxy.

Because the module is experimental, SWIG emits warning 524 when -rust is used. The warning can be suppressed with -w524 after the user has accepted the experimental status.

40.2 Additional command line options

The following Rust-specific command line option is available: 

-crate <name>  Set the Rust module/crate name used in generated comments

The generated Rust module directory normally uses the SWIG module name. As in the C# module, the generated name can also be customized from the %module directive. Use rustname to set the generated Rust module directory and crate to set the crate name used in generated Rust comments: 

%module(rustname="example_rust", crate="example_crate") example

When the rustname option is used, Rust code imports the generated directory name: 

mod example_rust;

40.3 Simple example

Given a C source file: 

int add(int a, int b) {
  return a + b;
}

and the following SWIG interface: 

%module example

%{
extern int add(int a, int b);
%}

extern int add(int a, int b);

Run SWIG and build the shared library: 

$ swig -rust example.i
$ cc -fPIC -c example.c example_wrap.c
$ cc -shared example.o example_wrap.o -o libexample.so

The generated example/mod.rs can then be used from Rust: 

mod example;

fn main() {
    let result = example::add(19, 23);
    assert_eq!(result, 42);
}

Compile and run the Rust program with the generated library on the library path: 

$ rustc runme.rs -L . -l dylib=example -o runme
$ LD_LIBRARY_PATH=. ./runme

For C++ code, use -c++ and compile the generated wrapper with a C++ compiler: 

$ swig -rust -c++ example.i
$ c++ -fPIC -c example.cxx example_wrap.cxx
$ c++ -shared example.o example_wrap.o -o libexample.so

The source tree contains runnable Rust examples under Examples/rust. After configuring and building SWIG with Rust enabled, run: 

$ make -C Examples/rust/simple check
$ make -C Examples/rust/class check
$ make -C Examples/rust/director check

The Rust test-suite can be run with make check-rust-test-suite. The Linux GitHub Actions workflow runs the Rust examples and test-suite in the default compiler mode and in C++11, C++14 and C++17 configurations. Rust remains marked experimental in CI, so these jobs are allowed to fail without failing the entire workflow while the target language is still under development.

40.4 Exception handling

Rust wrappers use the normal SWIG "throws" typemap mechanism for C++ exceptions. The Rust std_except.i library file installs default typemaps for the standard C++ exception hierarchy. When a C++ exception is caught by the generated wrapper, the Rust side receives a structured RustException: 

pub struct RustException {
    pub kind: RustExceptionKind,
    pub code: Option<&'static str>,
    pub type_name: String,
    pub message: String,
}

pub enum RustExceptionKind {
    BadCast,
    InvalidArgument,
    OutOfRange,
    Overflow,
    Runtime,
    Logic,
    Unknown,
}

RustExceptionKind is a compact target language category. The exact C++ exception type is kept in type_name, for example "std::out_of_range". RustException implements the standard Rust Display and Error traits.

By default the generated Rust API follows SWIG's traditional compatibility style: a plain function or method panics if C++ throws, while a generated try_ wrapper returns a Result: 

let value = example::read_value();       // panics on C++ exception
let value = example::try_read_value()?;  // returns Result<T, RustException>

match example::try_read_value() {
    Ok(value) => println!("{}", value),
    Err(error) => eprintln!("{}: {}", error.type_name, error.message),
}

For a Rust-first API, use the exceptionpolicy pragma: 

%pragma(rust) exceptionpolicy="result"

With this policy enabled, the plain function, method or constructor returns Result. The panic-on-exception compatibility wrapper is still generated with a panic_ prefix: 

let value = example::read_value()?;       // returns Result<T, RustException>
let value = example::panic_read_value();  // panics on C++ exception

let object = example::Widget::new()?;     // returns Result<Widget, RustException>
let object = example::Widget::panic_new();

The accepted policy values are "panic" and "result". The default is "panic".

The compact RustExceptionKind category enum can be customized for module-specific C++ exception types: 

%pragma(rust) exceptionkind="MyLibraryError=Logic"
%pragma(rust) exceptioncode="MyLibraryError=MY_LIBRARY_ERROR"

%typemap(throws, canthrow=1) MyLibraryError %{
  SWIG_RustSetPendingException("MyLibraryError", $1.what());
  return $null;
%}

When the generated Rust code receives "MyLibraryError", the RustException keeps type_name == "MyLibraryError", maps kind to RustExceptionKind::Logic, and stores code == Some(MY_LIBRARY_ERROR). Accepted category names are BadCast, InvalidArgument, OutOfRange, Overflow, Runtime, Logic and Unknown.

Director callbacks are protected by generated panic and error bridges. A Rust callback can panic, or it can call rust_director_error to report a structured callback error. The generated C++ director receives this as a director exception path instead of allowing a Rust panic to unwind through C++. If the C++ virtual method is declared noexcept, the generated director terminates rather than throwing through the non-throwing exception specification.

40.5 Ownership and nullability

Generated Rust proxy classes store C++ object pointers as std::ptr::NonNull<c_void> together with a RustOwnership value. A proxy object therefore cannot contain a null pointer. C++ pointer return values are represented as Option<T>, so a null C++ pointer becomes None. References and value returns are converted to non-null proxies and panic if the C++ wrapper unexpectedly returns null.

Owned proxies delete the wrapped C++ object from Drop. Borrowed proxies never delete the wrapped object. Generated proxy classes provide ownership(), is_owned(), is_borrowed(), borrow_proxy(), disown() and into_raw() to make the ownership transition explicit. Raw pointer construction remains unsafe and uses from_raw(), borrowed() or from_raw_owned(), each of which returns Option<T> to reject null pointers. 

let mut widget = example::Widget::new();
assert!(widget.is_owned());

let borrowed = widget.borrow_proxy();
assert!(borrowed.is_borrowed());

let raw = widget.into_raw();          // Rust no longer owns the C++ object
let widget = unsafe {
    example::Widget::from_raw(raw)
}.expect("expected non-null Widget pointer");

Function and method parameters use Rust references when SWIG can identify a wrapped proxy type. A C++ const T * parameter becomes Option<&T>, T * becomes Option<&mut T>, const T & becomes &T and T & becomes &mut T. The raw FFI layer still receives C pointers; the public proxy layer converts Rust references and None into the appropriate C ABI pointer. 

let mut object = example::Widget::new();

example::inspect(Some(&object));      // const Widget *
example::update(Some(&mut object));   // Widget *
example::use_ref(&object);            // const Widget &
example::mutate_ref(&mut object);     // Widget &

The public as_ptr() method exposes a read-only *const c_void for diagnostics or integration with lower-level code. Generated module internals use a crate-visible mutable pointer accessor for calling the raw C ABI wrappers. borrow_proxy() returns a lifetime-bound borrowed proxy view which dereferences to the proxy type and never frees the C++ object.

40.6 Standard library wrappers

The Rust standard library files follow the same broad organization as the C# library files while exposing Rust-shaped helper methods where possible. The library includes wrappers for strings, wide strings, string views, vectors, arrays, lists, deques, maps, sets, unordered maps, unordered sets, pairs, optional values, C data buffers, complex values, common STL support and smart pointers such as std::shared_ptr, std::unique_ptr, std::auto_ptr and Boost smart pointers.

Sequence-like containers such as std::vector, std::deque, std::list and std::array provide len(), is_empty(), indices(), iter() and IntoIterator for borrowed proxies in addition to the lower-level wrapped C++ methods. Associative containers such as std::map, std::set, std::unordered_map and std::unordered_set provide len(), is_empty() and borrowed iteration. Maps also provide keys(), values() and pair iteration. 

let values = example::IntVector::new();
values.push_back(10);
values.push_back(20);

assert_eq!(values.len(), 2);
assert!(!values.is_empty());

for value in &values {
    println!("{}", value);
}

let table = example::IntIntMap::new();
table.setitem(4, 40);
table.setitem(8, 80);

let key_sum: i32 = table.keys().sum();
let value_sum: i32 = table.values().sum();
let product_sum: i32 = table.iter().map(|(key, value)| key * value).sum();

assert_eq!(key_sum, 12);
assert_eq!(value_sum, 120);
assert_eq!(product_sum, 800);

The std_optional.i wrapper maps std::optional<T> to Option<T> for wrapped proxy values. A present optional object becomes Some(value); an empty optional becomes None.

40.7 C++ namespaces

The Rust module supports the common SWIG %nspace feature. When enabled, top-level C++ classes and enums in a namespace are emitted into matching Rust submodules below the generated module directory, in the same style that the C# module emits namespace-specific proxy files. 

%module example
%nspace;

%inline %{
namespace outer {
class Thing {
public:
  Thing(int value) : value_(value) {
  }
  int value() const {
    return value_;
  }
private:
  int value_;
};
}
%}

This generates files such as example/outer/mod.rs and example/outer/class_Thing.rs. Rust code can then use the namespaced proxy through the generated Rust module path: 

mod example;

fn main() {
    let thing = example::outer::Thing::new(7);
    assert_eq!(thing.value(), 7);
}

40.8 Pragmas

The Rust module supports %pragma(rust) customization points modelled after the C# module's pragma mechanism. The following pragmas customize the generated Rust module: 

%pragma(rust) moduleimports=%{
use std::os::raw::c_int as MyInt;
%}

%pragma(rust) modulecode=%{
pub const MODULE_MARKER: MyInt = 10;
%}

%pragma(rust) externattributes=%{
#[allow(improper_ctypes)]
%}

%pragma(rust) externcode=%{
  pub fn extra_c_symbol(input: MyInt) -> MyInt;
%}

%pragma(rust) classattributes=%{
#[repr(C)]
%}

%pragma(rust) classderive="Debug"

%pragma(rust) classcode=%{
  pub fn marker(&self) -> MyInt {
    MODULE_MARKER
  }
%}

%pragma(rust) visibility="pub"

%pragma(rust) exceptionpolicy="result"

%pragma(rust) exceptionkind="MyLibraryError=Logic"

%pragma(rust) exceptioncode="MyLibraryError=MY_LIBRARY_ERROR"

%pragma(rust) methodcase="preserve"

moduleimports and modulecode are emitted in module.rs. Public items emitted by modulecode are re-exported by mod.rs. moduleimports are also emitted where generated FFI and proxy code may need the same aliases. externattributes and externcode customize the generated extern "C" block in ffi.rs. classattributes, classderive and classcode are applied to generated proxy classes. visibility sets the visibility token used for generated proxy classes and constants.

exceptionpolicy selects whether plain generated wrappers panic on C++ exceptions ("panic") or return Result ("result"). exceptionkind maps a concrete C++ exception type name to a RustExceptionKind category. exceptioncode maps a concrete C++ exception type name to a stable string code stored in RustException::code and emits a public constant for that code. methodcase controls generated public free function and proxy method names; accepted values are "preserve" and "snake". The default is "preserve", so generated Rust names match the original C++ API unless the user explicitly asks SWIG to convert them.

40.9 Typemap customization

The Rust module uses Rust-specific typemap names for the same stages that C# customizes with imtype, cstype, csin, csout, csvarin, csvarout, csdirectorin and csdirectorout. The corresponding Rust names are imtype, rusttype, rustin, rustout, rustvarin, rustvarout, rustdirectorin and rustdirectorout. The Rust rustin and rustdirectorin typemaps support pre, post and terminator attributes for setup and cleanup code around generated calls.

Generated proxy classes can be customized with rustattributes, rustderive, rustbody, rusttraits and rustdrop typemaps. These correspond to the common C# pattern of injecting proxy attributes, proxy code, interface implementations and disposal customization, but emit Rust attributes, inherent impl methods, trait impls and drop hook code. 

%typemap(rustderive) Widget "Debug"
%typemap(rustbody) Widget %{
  pub fn marker(&self) -> i32 {
    7
  }
%}
%typemap(rusttraits) Widget %{
impl WidgetMarker for Widget {
  fn marker_value(&self) -> i32 {
    self.marker()
  }
}
%}
%typemap(rustdrop) Widget %{
  eprintln!("dropping Widget proxy");
%}

Some C# typemap attributes do not have direct Rust equivalents because Rust has different language concepts. For example, C# property attributes and explicit interface lists map to Rust pragmas and trait impl injection rather than to a single target-language attribute string. The Rust backend documents these as Rust-specific pragmas and typemaps rather than keeping C# names in the public customization surface.

40.10 Directors

Director support is enabled in the same way as other director capable target languages. Use the directors module option and apply the director feature to the C++ class whose virtual methods should be overridden from Rust: 

%module(directors="1") messages
%feature("director") Receiver;

%inline %{
class Receiver {
public:
  Receiver() : last(0) {
  }
  virtual ~Receiver() {
  }
  virtual void message(int value) {
    last = value;
  }
  int last;
};

inline void deliver(Receiver *receiver, int value) {
  receiver->message(value);
}
%}

For a class named Receiver, the generated Rust module contains an opaque Receiver proxy struct, callback type aliases such as ReceiverCallback0, method metadata arrays such as ReceiverDirectorMethodTypes0, a ReceiverDirector trait describing the callbacks, a raw Receiver::connect_director method, a typed Receiver::connect_director_trait helper and a ReceiverDirectorHandle helper which keeps the Rust callback object alive and disconnects the director when the handle is dropped. When %nspace is used, these director helpers are emitted next to the proxy type in the generated namespace module. 

mod messages;

use messages::{Receiver, ReceiverDirector, ReceiverDirectorOwned};
use std::os::raw::c_int;

struct MyReceiver {
    total: c_int,
}

impl ReceiverDirector for MyReceiver {
    fn message(&mut self, value: c_int) {
        self.total += value;
    }
}

fn main() {
    let mut receiver =
        ReceiverDirectorOwned::from_proxy(Receiver::new(), MyReceiver { total: 0 });

    messages::deliver(Some(receiver.as_proxy_mut()), 42);
    assert_eq!(receiver.director().total, 42);
}

The exact callback alias numbers follow the order of director methods emitted for the class. For example, the first director method in Receiver uses ReceiverCallback0. The raw connect_director method remains available for code that wants to provide callback function pointers directly, but ordinary Rust code should prefer the generated trait and owned helper types.

Director method arguments and return values are marshalled through Rust ctype, directorin, directorout, directorargout, rustdirectorin and rustdirectorout typemaps, following the same director typemap structure used by the C# module. Constructed Rust proxy objects own their C++ object by default and release it from Drop; use disown() or into_raw() when the C++ side takes ownership.

Director handles can be converted into a disowned guard with into_disowned(). This keeps the Rust callback state alive for C++ code that stores the director object and later allows explicit disconnection or recovery of the callback state. User director trait methods are safe Rust methods; generated extern callback shims contain the raw pointer reconstruction and panic/error bridge. If a callback needs to inspect a raw pointer argument such as void * or char *, that operation remains explicitly unsafe in the callback body. Protected virtual director methods are supported with the dirprot module option.

For callback-heavy APIs where a static C++ create method stores a director callback and returns a proxy object, SWIG also generates a Rust helper which owns both the returned proxy and the callback guard. This avoids dropping the Rust callback state while C++ still holds the director pointer: 

let mut service =
    MessageService::try_create_with_director(MyReceiver { total: 0 })?;

service.sendEnable(true);
service.sendValue("temperature".to_string(), 23);

assert!(service.director().enabled);
assert_eq!(service.director().last_id, "temperature");

let (service_proxy, callbacks) = service.into_parts();

The lower-level DirectorOwned and DirectorHandle APIs remain available for methods that do not follow this factory shape, for APIs that register more than one callback, or for code that needs explicit disconnect and disown control.
