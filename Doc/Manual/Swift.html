This chapter describes SWIG's support for Swift, Apple's compiled, statically typed language designed for safety and performance. The Swift backend is experimental - basic functionality works but coverage is not yet on a par with the fully supported language modules.

SWIG generates two files when targeting Swift: a C++ wrapper (module_wrap.cxx) that exposes C bridge functions, and a Swift source file (module.swift) containing the proxy classes and free functions that Swift code imports directly. No shared-library loading step is required beyond the standard framework or dylib linking performed by the Swift build system.

40.1 Preliminaries

You will need:

  • SWIG 4.5.0 or later built with Swift support (the -swift flag is marked Experimental).
  • A C++ compiler for the generated wrapper (any compiler supported by the target platform).
  • Swift 5.7 or later (Xcode 14+ on Apple platforms, or the open-source Swift toolchain on Linux).

Because the Swift backend is experimental, SWIG emits warning 524 whenever it is invoked. Suppress it with -w524 once you are aware of the status.

40.2 Running SWIG

Invoke SWIG with the -swift flag and C++ mode: 

swig -swift -c++ -w524 example.i

The generated files are placed in the current directory by default. Use -outdir <dir> to redirect the Swift proxy file: 

swig -swift -c++ -w524 -outdir swift/ -o swift/example_wrap.cxx example.i

The module declaration in the interface file sets the Swift module name: 

%module mylib

This produces mylib_wrap.cxx, mylib_wrap.h, and mylib.swift.

40.2.1 Swift command line options

The following options are available with -swift: 

-outfile <file>  Override the name of the generated .swift file
-autorename      Rename wrapped methods to Swift's lowerCamelCase convention
-doxygen         Convert C++ Doxygen comments to Swift documentation comments

By default a wrapped method keeps its C++ name, so GetVersion() stays GetVersion() in Swift. With -autorename the leading capital is lower cased to follow Swift naming conventions, so GetVersion() becomes getVersion(). Types, enumerations and enumeration cases are never renamed. When -autorename and -doxygen are combined, references to renamed functions inside the translated documentation comments are rewritten to match the generated Swift names.

40.2.2 Compiling the wrapper

Compile the C++ wrapper alongside your C++ library sources to produce a shared library or framework. The Swift proxy file is compiled as part of the Swift module that imports the library: 

# Compile the C++ wrapper
clang++ -fPIC -std=c++17 -c mylib_wrap.cxx -o mylib_wrap.o

# Link with your library
clang++ -shared mylib_wrap.o mylib.o -o libmylib.so

On Apple platforms you would build a framework or dylib and use a module map or @_implementationOnly import to expose the C bridge header to Swift.

Complete, runnable examples are provided under Examples/swift/simple (free functions and a global variable) and Examples/swift/class (C++ classes with inheritance and virtual dispatch); each builds the wrapper and runs a runme.swift with make check. Both compile the generated proxy against the wrapper using the C bridging header (-import-objc-header module_wrap.h).

40.3 A basic tour of C/C++ wrapping

Two conventions apply throughout the generated Swift API and are worth keeping in mind while reading the examples below:

  • Every wrapped function, method and initializer is declared throws, because any C++ call may raise an exception that is propagated to Swift (see Exception handling). Call sites therefore use try.
  • Parameters carry argument labels taken from the C++ parameter names, so a C++ add(int x, int y) is called as add(x: 3, y: 4) from Swift.

40.3.1 Functions

A C++ free function is wrapped as a Swift global function. C int maps to Swift Int32 and double to Double: 

%module example
%{
#include "example.h"
%}

int add(int x, int y);
double square(double x);

From Swift: 

import example

let result = try add(x: 3, y: 4)   // 7
let s = try square(x: 2.5)         // 6.25

40.3.2 C++ classes

A C++ class becomes a Swift class. The proxy class owns the underlying C++ object via an opaque swigCPtr and manages its lifetime through swigCMemOwn. The SWIG-generated delete() method releases the object immediately when called; otherwise the object is freed when Swift ARC collects the proxy. 

%module example
%{
#include "Point.h"
%}

class Point {
public:
  Point(double x, double y);
  double getX() const;
  double getY() const;
  void   setX(double x);
  void   setY(double y);
};

import example

let p = try Point(x: 1.0, y: 2.0)
print(try p.getX())   // 1.0
try p.setY(y: 5.0)
print(try p.getY())   // 5.0

40.3.3 Enumerations

C++ enumerations are wrapped as Swift enumerations with Int32 raw values: 

enum Color { RED, GREEN, BLUE };

let c = Color.RED
print(c.rawValue)   // 0

40.3.4 Inheritance

Single inheritance is fully supported. The Swift proxy class mirrors the C++ inheritance hierarchy. Multiple inheritance is not supported; only the first base class is used and a warning (864) is emitted for each additional base.

Deriving from a base class that was wrapped in a separate module and pulled in with %import is also supported. A constructor inherited across the module boundary is generated with Swift's override keyword so the derived proxy compiles against the imported base.

40.3.5 Nested classes and structs

C++ nested classes and structs are wrapped as true Swift inner types rather than being hoisted to module scope. A C++ nested class such as Outer::Config becomes Outer.Config in Swift: 

class Outer {
public:
  struct Config {
    int width;
    int height;
  };
  void apply(const Config& c);
};

let cfg = try Outer.Config()
let w = try cfg.GetWidth()    // accessor method, see note below
let obj = try Outer()
try obj.apply(c: cfg)

Public member variables are currently exposed through generated accessor methods (GetWidth() here) rather than as assignable Swift properties - see Known limitations.

This is different from the Python and Ruby backends, which use %feature("flatnested") to hoist nested types to module scope. The Swift backend does not apply flatnested; nested types follow Swift's natural inner-type idiom.

40.4 Exception handling

C++ exceptions thrown inside wrapper functions are caught by SWIG and propagated to Swift as a thrown Error. Every wrapped C++ function, method and initializer is therefore declared throws, and the swigCheckException() helper emitted at the top of the generated .swift file is called after each underlying C call to rethrow any pending exception: 

do {
  try someWrappedFunction()
} catch let error as NSError {
  print("C++ threw: \(error.localizedDescription)")
}

Untyped exceptions (the C++ standard exceptions such as std::exception, director exceptions and unknown C++ exceptions) are surfaced as a Foundation NSError, using the SWIG module name as the error domain and the C++ message as localizedDescription. This matches how the Java and C# modules map these cases to standard language exceptions and keeps the generated public API free of SWIG specific error types.

A function that returns a pointer or a std::shared_ptr maps to a Swift Optional and returns nil when the C++ side returns a null pointer. If the C++ contract is "return a valid object or fail", you can make such a function return a non-optional value that throws on null instead - see %feature("swift:nonnull").

40.5 Directors - callbacks into Swift

Directors allow Swift code to subclass a C++ class and override its virtual methods. Enable directors for a class with: 

%module(directors="1") example

%feature("director") Logger;

%{
#include "Logger.h"
%}

class Logger {
public:
  virtual ~Logger() {}
  virtual void log(const std::string& message) = 0;
};

When %feature("director") is applied, SWIG marks the Swift proxy class open so that Swift code can subclass it directly. Internally SWIG generates a C++ class SwigDirector_Logger : public Logger whose virtual methods dispatch back into Swift via stored function-pointer callbacks, but this C++ detail is not visible to Swift callers. The proxy class Logger is all you see and all you subclass: 

class MyLogger: Logger {
  override func log(message: String) throws {
    print("[LOG] \(message)")
  }
}

let logger = try MyLogger()
// pass logger to C++ as a Logger*

Virtual methods keep their C++ names in Swift. They are declared open func inside the proxy class so subclasses can override them with the normal Swift override keyword. All of them are marked throws so Swift exceptions propagate back to C++ as a Swig::DirectorException via a thread-local pending-exception state. Swift ARC keeps the proxy object alive as long as C++ holds the back-pointer, so no extra retention wrapper is needed.

40.6 Memory management

Every SWIG proxy class stores the C++ object behind an opaque pointer (swigCPtr) and a Boolean ownership flag (swigCMemOwn). When swigCMemOwn is true, the proxy's deinitializer calls delete on the underlying C++ object. When SWIG passes an object into C++ code that takes ownership (for example via a factory function that accepts a raw pointer), the proxy's ownership flag is cleared to prevent a double-free.

Swift ARC (Automatic Reference Counting) manages the proxy lifetime. As long as at least one Swift reference to the proxy is live, the underlying C++ object is not freed. This is different from Java and Python, where the garbage collector may collect proxies that C++ still holds via a director back-pointer; Swift ARC's deterministic retain/release makes such extra retention unnecessary.

40.7 Customizing the generated code

Two directives let you adjust the generated Swift without writing typemaps.

40.7.1 Non-optional pointer returns

By default a function that returns a pointer or a std::shared_ptr maps to a Swift Optional, because a C++ pointer may be null: 

struct Widget {
  /* ... */
};
Widget *makeWidget();

// Returns Widget? - nil when the C++ pointer is null
public func makeWidget() throws -> Widget? {
    let jresult = _wrap_makeWidget()
    try swigCheckException()
    guard let ptr = jresult else { return nil }
    return Widget(ptr, true)
}

When the C++ contract is "return a valid object or fail", forcing callers to unwrap an Optional that is never meant to be nil is noise. Apply %feature("swift:nonnull") to the declaration to return a non-optional type that throws when the pointer is unexpectedly null instead: 

%feature("swift:nonnull") makeWidget;
Widget *makeWidget();

// Returns a non-optional Widget; throws instead of returning nil
public func makeWidget() throws -> Widget {
    let jresult = _wrap_makeWidget()
    try swigCheckException()
    guard let ptr = jresult else {
        throw NSError(domain: "example", code: 1,
                      userInfo: [NSLocalizedDescriptionKey: "makeWidget unexpectedly returned a null pointer"])
    }
    return Widget(ptr, true)
}

The feature follows SWIG's usual %feature scoping, so it can be applied to a single declaration, a class, or globally. It only affects functions returning a wrapped pointer or shared pointer; value, scalar and void returns are unchanged.

40.7.2 Injecting module level Swift code

Use the %pragma(swift) directive to add imports or top level Swift code to the generated .swift file. Within these blocks $module expands to the SWIG module name, which is useful for extending the module's own types. Use moduleimports for import statements and modulecode for declarations: 

%pragma(swift) moduleimports = "import Foundation"
%pragma(swift) modulecode = %{
extension $module {
  // module level Swift helpers
}
%}

40.8 Typemaps

Typemaps let you change how a C/C++ type crosses into Swift. Before writing one it helps to understand that every wrapped call passes through two layers:

  • The C bridge in module_wrap.cxx (declared in module_wrap.h): for each wrapped entity SWIG emits an extern "C" function such as _wrap_square whose signature uses plain C ABI types. This is the only surface Swift links against.
  • The Swift proxy in module.swift: the public function or method your Swift code calls, which forwards to the C bridge function (imported through the generated bridging header).
  Swift call site
        |   swifttype          (the public Swift type, e.g. Int32, Widget?)
        v
  Swift proxy func   module.swift        swiftin / swiftout
        |   imtype             (Swift type of the imported C function)
        v
  C bridge func _wrap_*   module_wrap.cxx / .h    ctype, in / out
        |   C++ types
        v
  your C++ function

Each typemap slot fills one role in that pipeline:

TypemapLayerWhat it controls
ctypeC bridgeThe C ABI type used in the _wrap_* function signature (for example int, or void * for a wrapped object).
imtypeC bridge / SwiftThe Swift type the imported bridge function presents; must be ABI compatible with ctype (Int32 for int, UnsafeMutableRawPointer? for void *).
swifttypeSwiftThe type shown in the proxy's public API - what your Swift code sees and passes.
inC bridgeInside _wrap_*, convert the incoming C value ($input) to the C++ value ($1).
outC bridgeInside _wrap_*, convert the C++ result ($1) to the C return value ($result).
swiftinSwiftThe Swift expression that turns a proxy argument ($swiftinput) into the value passed to the bridge call.
swiftoutSwiftThe Swift body that calls the bridge ($imcall), checks for a pending exception, and converts the result to swifttype.
swiftclassmodifiersSwiftAccess modifiers placed before class in a proxy declaration (public, or open for director classes).
swiftbody / swiftbody_derivedSwiftThe stored properties, initializer and swigGetCPtr emitted inside a proxy class (the derived variant is used for subclasses).
swiftdestruct / swiftdestruct_derivedSwiftThe deinit that releases the C++ object when the proxy owns it.
directorin / directoroutC++Director only: convert a callback parameter from C++ to C, and convert the Swift override's return value from C back to C++.
swiftdirectorin / swiftdirectoroutSwiftDirector only: the Swift side of the same conversions, inside the dispatch closure that calls your override.

Standard typemaps for bool, the integer and floating point types and void ship in Lib/swift/swiftprimitives.swg and wrapped class types in Lib/swift/swiftswigtype.swg; both are included automatically. The following library files add further support and are included from your interface file as needed: 

%include <swift/std_string.i>        // std::string <-> Swift String
%include <swift/std_vector.i>        // std::vector<T> as a Swift collection
%include <swift/std_shared_ptr.i>    // std::shared_ptr<T>
%include <swift/boost_shared_ptr.i>  // boost::shared_ptr<T>

Each %template instantiation of std::vector<T> produces a Swift proxy class that conforms to RandomAccessCollection, so the wrapped vector can be iterated and indexed like a native Swift collection: 

%include <swift/std_vector.i>
%template(IntVector) std::vector<int>;

let v = try IntVector()
try v.push_back(x: 10)
try v.push_back(x: 20)
for value in v {        // RandomAccessCollection conformance
  print(value)
}
print(v[0])             // 10

40.8.1 A worked example

Take a single C++ function: 

int square(int x);

SWIG emits this C bridge function into module_wrap.cxx. The signature type int comes from ctype, arg1 = jarg1 is the in typemap, and jresult = result is the out typemap: 

SWIGEXPORT int _wrap_square(int jarg1) {
  int jresult = 0;
  int arg1;
  int result;
  arg1 = jarg1;                  // (in)  C value  -> C++ value
  result = (int)square(arg1);
  jresult = result;              // (out) C++ value -> C value
  return jresult;
}

and this proxy function into module.swift. The public types Int32 come from swifttype, the imported _wrap_square takes/returns imtype, the argument x is the swiftin expression, and the whole body is the swiftout typemap: 

public func square(x: Int32) throws -> Int32 {
    let jresult = _wrap_square(x)   // swiftin supplies the argument 'x'
    try swigCheckException()        // }
    return jresult                  // } swiftout body
}

For a wrapped class type the same slots apply but the bridge type is an opaque handle: ctype is void *, imtype is UnsafeMutableRawPointer?, swifttype is the proxy class name, swiftin passes the object's swigCPtr, and swiftout wraps the returned handle back into a proxy object.

40.8.2 Director typemaps

For a director class the data flows the opposite way - C++ calls into Swift - so two extra pairs of typemaps are used. When a C++ virtual method fires, directorin converts each C++ argument to the C bridge type and swiftdirectorin converts it to the Swift value handed to your override; on return, swiftdirectorout converts the Swift result back to the C bridge type and directorout converts it back to the C++ type the virtual method must return. The primitive director typemaps are pass-through; the wrapped-type ones marshal the void * handle to and from a proxy object.

40.8.3 Defining your own typemaps

To wrap a type that has no built in mapping, supply the slots for the layers it crosses. For example, to pass a small C++ value type as a plain Swift Double rather than a proxy object, map its bridge type to double and unpack it in the in / out typemaps: 

struct Celsius {
  double degrees;
};

%typemap(ctype)     Celsius "double"
%typemap(imtype)    Celsius "Double"
%typemap(swifttype) Celsius "Double"
%typemap(in)        Celsius %{ $1 = Celsius{$input}; %}
%typemap(out)       Celsius %{ $result = $1.degrees; %}
%typemap(swiftin)   Celsius "$swiftinput"
%typemap(swiftout)  Celsius %{
    let jresult = $imcall
    try swigCheckException()
    return jresult
%}

A function declared Celsius freezing(); then surfaces in Swift as public func freezing() throws -> Double, and Celsius parameters likewise appear as Double, with no proxy class generated for the type.

40.9 Known limitations

As an experimental backend the Swift module covers the common cases but not yet the full feature set of the mature language modules. The main areas not yet supported are:

  • Multiple inheritance - Swift is single inheritance, so only the first base class is wrapped and warning 864 is issued for each additional base.
  • C++ exception specifications - functions with a throw(T) specification or %catches have no throws typemap (warning 468); exceptions are still propagated, but the typed specification is not honoured.
  • Member (pointer to member) function pointers are not wrapped.
  • Arrays of enumerations - a fixed C array whose element type is an enum (for example Colour finger[5]) is not marshalled: the array accessor uses the generic SWIGTYPE handling, which expects a wrapped class rather than the Swift enum representation. A scalar, pointer or reference to an enum is handled.
  • STL beyond std::string, std::vector and the shared pointer wrappers - std::map, std::pair, std::deque, std::unique_ptr, cdata and the general typemaps.i library are not yet provided.
  • Directors - the common cases work, including abstract classes, inheritance, overloads, enums, template and nested director bases, conversion operators, inherited (using) constructors, a C++ base with no default constructor (each accessible base constructor is forwarded through the director), and a director base defined later in the translation unit. The remaining gaps follow the general limitations above: a director method with a C++ exception specification (warning 468), a reference or rvalue-reference qualified virtual, or a move-only parameter is not wrapped. A virtual method that a Swift subclass does not override is not forwarded to the C++ base implementation (the generated open func stub runs instead).
  • Public member variables are exposed read only, through a generated GetName() accessor rather than an assignable property (no setter).
  • %apply based multi-argument typemaps such as the (char *STRING, size_t LENGTH) binary string pattern and DYNAMIC casts are not defined (warning 453).
  • Varargs (...) are not supported.
  • C wrapping - the backend targets C++; run SWIG with -c++ (the generated wrapper uses C++ even for a C interface).
  • Smaller gaps - a constant whose value has no Swift equivalent is omitted from the generated module: a boolean-valued expression (#define M (A>B), 1 && 0, a chained comparison), a sizeof or cast expression, or an unexpanded function-style macro. Simple integer, floating point, string and boolean literal constants are wrapped.

The Swift test suite records the individual unsupported test cases in the FAILING_CPP_TESTS and FAILING_C_TESTS lists of Examples/test-suite/swift/Makefile.in, where they are grouped by cause and each group is annotated with the limitation above that explains it.
